# توثيق نظام منشئ أنواع المحتوى (Content Type Builder System)

تم تصميم وبناء وحدة "منشئ أنواع المحتوى" (Content Type Builder) كأداة بصرية مرنة لإدارة قوالب البيانات وتمديدها بحقول مخصصة بالكامل تماثل الأنظمة الرائدة في الـ Headless CMS، وذلك في الصفحات [Pages/ContentTypes.cshtml](file:///e:/Projects/TalaPress/Pages/ContentTypes.cshtml) و [Pages/ContentTypes.cshtml.cs](file:///e:/Projects/TalaPress/Pages/ContentTypes.cshtml.cs).

---

## 1. الفلسفة الأساسية للنظام (Core Philosophy)

يعتمد نظام TalaPress على أسلوب هجين (Hybrid Approach):
1. **الحقول النظامية المشتركة (System/Core Fields)**: تشترك جميع قوالب المحتوى في مجموعة أساسية من الحقول المضمنة مسبقاً في جدول `dbo.Content` (مثل العنوان، الرابط الفرعي، الملخص، المحتوى، الصورة البارزة، القسم، وتاريخ النشر).
2. **حقول محركات البحث المضمنة (SEO Fields)**: تتوفر حقول تهيئة محركات البحث تلقائياً لجميع أنواع المحتوى (مثل عنوان سيو، وصف سيو، الكلمات المفتاحية، ورابط الكانونيكال).
3. **الحقول المخصصة الإضافية (Custom Fields)**: يتيح المنشئ للمشرفين إضافة وتخصيص حقول إضافية غير محدودة خاصة بطبيعة العمل (مثل السعر، تاريخ الفعالية، جهة المصدر، إلخ) لكل قالب بشكل مستقل ودون كتابة أي كود برمجي.

## 2. هيكل قاعدة البيانات (Database Schema)

يعتمد الموديول على جدولين رئيسيين في قاعدة البيانات للتحكم في قوالب البيانات والحقول المرتبطة بها:

### أ. جدول قوالب المحتوى (`dbo.ContentTypes`)
يخزن البيانات التعريفية لكل نوع محتوى.
* **الحقول**:
  * `Id` (BIGINT, IDENTITY, PK): المعرف الفريد لنوع المحتوى.
  * `Name` (NVARCHAR(100)): الاسم بالعربية (مثال: خبر، حدث، منتج).
  * `Name_En` (NVARCHAR(100)): الاسم بالإنجليزية (مثال: News, Event, Product).
  * `Description` / `Description_En` (NVARCHAR(1000), Nullable): وصف القالب باللغتين.
  * `IconValue` (NVARCHAR(500)): أيقونة القالب (أكواد Bootstrap Icons مثل `bi-newspaper`).
  * `IsSystem` (BIT, Default 0): يحدد ما إذا كان القالب أساسياً من النظام (لا يمكن حذفه).
  * `IsActive` (BIT, Default 1): يحدد ما إذا كان القالب نشطاً ومتاحاً للاستخدام.
  * `CreatedAt` (DATETIME2) / `UpdatedAt` (DATETIME2, Nullable).

### ب. جدول حقول أنواع المحتوى (`dbo.ContentTypeFields`)
يخزن الحقول المخصصة الإضافية التي يقوم المستخدمون بتعريفها لكل قالب.
* **الحقول**:
  * `Id` (BIGINT, IDENTITY, PK): معرف الحقل الفريد.
  * `ContentTypeId` (BIGINT, FK -> dbo.ContentTypes): معرف قالب المحتوى المرتبط بالحقل.
  * `FieldName` (NVARCHAR(100)): الاسم البرمجي الفريد للحقل (مثال: `news_source` أو `price`).
  * `FieldType` (NVARCHAR(50)): نوع الحقل (مثل: Text, RichText, Number, Select, Image, Gallery, CardList).
  * `Label` / `Label_En` (NVARCHAR(250)): التسمية التوضيحية الظاهرة باللغتين.
  * `Placeholder` / `Placeholder_En` (NVARCHAR(500), Nullable): نص التلميح التوجيهي للمحرر باللغتين.
  * `HelpText` / `HelpText_En` (NVARCHAR(1000), Nullable): النص الإرشادي والمساعد الظاهر تحت الحقل باللغتين.
  * `IsRequired` (BIT): يحدد ما إذا كان إدخال الحقل إجبارياً عند إنشاء المقال.
  * `IsTranslatable` (BIT): يحدد ما إذا كان الحقل يقبل الترجمة المتعددة (للغات المختلفة).
  * `DefaultValue` (NVARCHAR(MAX), Nullable): القيمة الافتراضية للحقل.
  * `OptionsJson` (NVARCHAR(MAX), Nullable): خيارات القائمة المنسدلة في حال كان نوع الحقل `Select` أو `MultiSelect`. يدعم النظام ثلاثة أنماط: خيارات ثابتة، خيارات ديناميكية، وخيارات ناتجة من `Query Builder`.
  * `SortOrder` (INT): ترتيب الحقل الفعلي في النموذج.
  * `IsActive` (BIT): حالة الحقل (نشط / معطل).
  * `CreatedAt` (DATETIME2).

## 3. تصميم واجهة المستخدم وتجربة الاستخدام (UI/UX Design)

تتبع صفحة العمل أسلوب **بيئة العمل المندمجة (Split-Screen Workspace)**:
* **لوحة القوالب الجانبية**: تعرض قائمة بجميع أنواع المحتوى المتوفرة، مع تمييز القوالب الأساسية (System) ببادرة "أساسي"، وإظهار أزرار سريعة للحذف (للقوالب المخصصة فقط) والنسخ المتطابق (Clone).
* **لوحة منشئ الحقول البصري (Visual Builder Panel)**: 
  * **بيانات القالب التعريفية**: نموذج لإدخال وتعديل الاسم، الوصف، الأيقونة، وحالة التنشيط.
  * **الحقول الأساسية للنظام (1. Core Fields)**: قسم علوي يعرض كافة حقول النظام الافتراضية بشكل بصري مغلق وقابل للقراءة فقط (رمادي اللون) ولا يمكن سحبه أو تعديله أو حذفه، ليؤكد للمستخدم أنها مضمنة تلقائياً.
  * **منطقة الحقول المخصصة (2. Custom Fields Workspace)**: مساحة عمل ذكية تفاعلية مدعومة بـ **`SortableJS`** تسمح للمستخدم بإعادة ترتيب الحقول المخصصة بالسحب والإفلات.
  * **حقول محركات البحث (SEO Fields)**: تم فصل حقول SEO عن مساحة السحب الرئيسية. تظهر دائماً أسفل نموذج نوع المحتوى داخل مجموعة ثابتة بإطار مستقل وغير قابلة للطي، بينما تظهر في نموذج إدخال المحتوى أسفل الصفحة داخل إطار قابل للطي `Accordion`.

### صندوق إعداد الحقل المخصص (Custom Attribute Modal)
* عند إضافة أو تعديل حقل، يظهر صندوق حوار متكامل للتحكم بجميع تفاصيله وخصائصه.
* حقل **الاسم البرمجي للحقل** يجب أن يكون بالإنجليزية فقط، يبدأ بحرف، ويسمح فقط بالأحرف الإنجليزية والأرقام والشرطة السفلية `_`. يتم التحقق من ذلك في الواجهة وفي السيرفر.
* حقلا **Label بالعربية** و **Label English** يظهران دائماً معاً بغض النظر عن لغة الواجهة، وكلاهما مطلوب لأنهما ترجمة مباشرة للحقل المخصص.
* يشتمل على **منشئ خيارات القائمة (Options Builder)**: يظهر فقط عند اختيار نوع حقل `Select` أو `MultiSelect` ويدعم أنماط القيم التالية:
  * `Static`: إدخال الخيارات يدوياً.
  * `Dynamic`: توليد الخيارات من مصدر محدد مثل التصنيفات أو المستخدمين.
  * `Query`: توليد الخيارات عبر `Query Builder` مرئي يعتمد على جداول وأعمدة قاعدة البيانات.

## 4. أنماط خيارات Select / MultiSelect

عند اختيار نوع الحقل `Select` أو `MultiSelect` يظهر قسم `إعداد خيارات القائمة المنسدلة`، وفيه حقل `نوع القيمة`.

### 4.1 Static / ثابتة

النمط الافتراضي لإدخال الخيارات يدوياً.

كل خيار يحتوي على:

* `value`: الكود أو القيمة المخزنة.
* `label`: الاسم بالعربية.
* `labelEn`: الاسم بالإنجليزية.

مثال:

```json
[
  {
    "value": "breaking",
    "label": "عاجل",
    "labelEn": "Breaking"
  }
]
```

### 4.2 Dynamic / ديناميكية

يستخدم هذا النمط لتوليد الخيارات من مصدر معروف داخل النظام.

المصادر الحالية:

* `category`: اختيار تصنيف رئيسي، ثم توليد الخيارات من التصنيفات الفرعية التابعة له.
* `user`: اختيار دور `Role`، ثم توليد الخيارات من المستخدمين المرتبطين بهذا الدور.

في حالة التصنيفات:

* `value` = رقم التصنيف الفرعي.
* `label` = اسم التصنيف بالعربية.
* `labelEn` = اسم التصنيف بالإنجليزية إن وجد.

في حالة المستخدمين:

* `value` = رقم المستخدم.
* `label` = الاسم الكامل أو اسم المستخدم.
* `labelEn` = نفس الاسم حالياً.

شكل التخزين داخل `OptionsJson`:

```json
{
  "mode": "dynamic",
  "source": "category",
  "sourceId": 1,
  "options": []
}
```

أو:

```json
{
  "mode": "dynamic",
  "source": "user",
  "sourceId": 2,
  "options": []
}
```

عند عرض نموذج إدخال المحتوى يتم حل الخيارات من قاعدة البيانات في وقت العرض.

### 4.3 Query / استعلام

هذا النمط يستخدم `Query Builder` مرئي لتوليد خيارات القائمة من جداول وأعمدة قاعدة بيانات الموقع.

عند اختيار `Query` يظهر:

* قائمة منسدلة بأسماء الجداول الموجودة في قاعدة البيانات ضمن `dbo`.
* عند اختيار جدول، يتم تحميل أعمدته في كل قاعدة `Rule`.
* يمكن تحديد المنطق العام `AND` أو `OR`.
* يمكن إضافة أكثر من Rule.
* كل Rule يحتوي على:
  * اسم العمود من الجدول المختار.
  * العملية باللغة الإنجليزية.
  * قيمة المقارنة.

العمليات المدعومة:

* `Equals`
* `DoesNotEqual`
* `Contains`
* `StartsWith`
* `EndsWith`
* `GreaterThan`
* `LessThan`
* `IsBlank`
* `IsNotBlank`
* `In`

بعد الضغط على `Execute`:

* يتم تنفيذ الاستعلام بطريقة آمنة بدون السماح بكتابة SQL يدوي.
* يتم توليد خيارات القائمة من النتائج.
* `value` يعتمد على عمود `Id` إن وجد، وإلا أول عمود في الجدول.
* `label` يتم اختياره بذكاء من أعمدة مثل `Title`, `Name`, `Label`, `FullName`, `Username`.
* `labelEn` يتم اختياره من النسخ الإنجليزية مثل `Title_En`, `Name_En`, `Label_En` إن وجدت.

شكل التخزين داخل `OptionsJson`:

```json
{
  "mode": "query",
  "query": {
    "tableName": "Content",
    "logic": "AND",
    "rules": [
      {
        "field": "Status",
        "operator": "Equals",
        "value": "Published"
      }
    ],
    "groups": []
  },
  "options": [
    {
      "value": "1",
      "label": "عنوان عربي",
      "labelEn": "English Title"
    }
  ]
}
```

> ملاحظة: `Query` ليس نوع حقل مستقل. هو نمط لتوليد خيارات `Select` و `MultiSelect` فقط.

## 4.4 CardList / قائمة بطاقات

نوع حقل `CardList` يُستخدم لبناء أقسام ببطاقات متكررة (مثل: أنواع الكفالات، خطوات العملية، ميزات الخدمة).

### بنية التخزين داخل `CustomFieldsJson`

```json
{
  "sponsorship_types": {
    "sectionTitle": "أنواع الكفالات",
    "sectionTitleEn": "Sponsorship Types",
    "sectionSubtitle": "ما الذي يميزنا؟",
    "sectionSubtitleEn": "What makes us special?",
    "sectionSummary": "اختر نوع الكفالة المناسب",
    "sectionSummaryEn": "Choose the right sponsorship type",
    "items": [
      {
        "title": "كفالة يتيم",
        "titleEn": "Orphan Sponsorship",
        "subtitle": "أساس تعاملنا مع المجتمع",
        "subtitleEn": "How we engage society",
        "description": "دعم شامل لليتيم",
        "descriptionEn": "Full support for an orphan",
        "mediaType": "icon",
        "icon": "bi-heart-fill",
        "image": "",
        "link": "/sponsorship/orphan",
        "featured": false
      }
    ]
  }
}
```

### حقول كل عنصر

| الحقل | الوصف |
|-------|--------|
| `sectionTitle` / `sectionTitleEn` | عنوان القسم |
| `sectionSubtitle` / `sectionSubtitleEn` | العنوان الفرعي للقسم |
| `sectionSummary` / `sectionSummaryEn` | ملخص اختياري للقسم |
| `items[].title` / `titleEn` | عنوان البطاقة |
| `items[].subtitle` / `subtitleEn` | العنوان الفرعي (مفيد للبطاقة المميزة) |
| `items[].description` / `descriptionEn` | وصف اختياري |
| `items[].mediaType` | `icon` أو `image` — يُخزَّن أحدهما فقط |
| `items[].icon` | أيقونة Bootstrap عند `mediaType=icon` |
| `items[].image` | صورة مرفوعة عند `mediaType=image` |
| `items[].link` | رابط اختياري |
| `items[].featured` | بطاقة مميزة واحدة فقط لكل قائمة |

> في المحرر: حقول العربية/الإنجليزية في تبويبات منفصلة؛ اختيار أيقونة **أو** صورة؛ مفتاح «مميز» يلغي تمييز البطاقات الأخرى تلقائياً.

## 5. البنية البرمجية الخلفية والتحقق (Backend Security & Validation)

تم بناء كافة العمليات بمرونة وأمان عاليين في الـ PageModel:
1. **حماية الصلاحيات (Claims Authorization Guards)**:
   * تفرض الصفحة صلاحية `ContentType.View` لعرض الصفحة والقوالب.
   * تفرض صلاحية `ContentType.Create` عند إنشاء أو استنساخ قالب.
   * تفرض صلاحية `ContentType.Edit` لحفظ التغييرات وإعادة الترتيب.
   * تفرض صلاحية `ContentType.Delete` لحذف أي قالب.
2. **سلامة المزامنة والعمليات الذرية (SQL Transaction Safety)**:
   * عند حفظ التغييرات، يتم استلام الحقول المخصصة كمصفوفة JSON من الواجهة الأمامية.
   * يقوم النظام داخل **SqlTransaction** بمسح الحقول القديمة وإعادة تعيين الحقول الجديدة وفقاً للترتيب والبيانات المحدثة لحظياً لتفادي أي تضارب أو تجزئة للبيانات في حال حدوث خطأ ما.
3. **حماية تكامل قوالب النظام (System Type Guards)**:
   * يُمنع المشرفون نهائياً من حذف القوالب الأساسية الـ 9 المضمنة مسبقاً (News, Event, Page, Gallery, Team, Case Study, Testimonial, Service, Project)، ولكن يُسمح لهم بتمديدها بحقول إضافية أو استنساخها بالكامل.
4. **قائمة الأسماء المحجوزة (Reserved Names Blocklist)**:
   * يقوم السيرفر بالتحقق من الاسم البرمجي المخصص (`FieldName`) ومنع حفظه إذا كان يطابق أحد الحقول النظامية المحجوزة (مثل `Title`, `Slug`, `CategoryId`) لتلافي مشاكل التداخل البرمجي.
5. **التحقق من صيغة الاسم البرمجي**:
   * يجب أن يطابق الاسم البرمجي النمط `^[A-Za-z][A-Za-z0-9_]*$`.
   * يمنع النظام الأسماء العربية، المسافات، الرموز الخاصة، أو الأسماء التي تبدأ برقم.
6. **تنفيذ Query Builder بشكل آمن**:
   * يتم تحميل أسماء الجداول والأعمدة من `INFORMATION_SCHEMA`.
   * لا يسمح للمستخدم بإدخال اسم جدول أو عمود يدوي خارج القائمة.
   * يتم بناء شروط الاستعلام باستخدام `SqlParameter` لمنع SQL Injection.
   * يتم التحقق من وجود الجدول والعمود قبل التنفيذ.
7. **النسخ المتطابق (Clone Feature)**:
   * يتيح زر "النسخ" استنساخ قالب بالكامل ونقل جميع الحقول المخصصة المحددة داخله إلى قالب جديد باسم `${origName} - نسخة` بضغطة زر واحدة.

## 6. انعكاس الخيارات على نماذج الإدخال والمعاينة والـ API

### نموذج إدخال المحتوى

عند إنشاء أو تعديل محتوى، يتم حل خيارات `Select` و `MultiSelect` قبل عرض النموذج:

* إذا كانت الخيارات `Static`، تستخدم مباشرة من `OptionsJson`.
* إذا كانت `Dynamic`، يتم تحميلها من قاعدة البيانات بناءً على المصدر المحدد.
* إذا كانت `Query`، يتم استخدام الخيارات الناتجة من الاستعلام والمخزنة في `OptionsJson`.

### المعاينة

تم تعديل المعاينات بحيث لا تعرض القيم الخام فقط، بل تعرض اسم الخيار أو الكابشن:

* المودال داخل صفحة المحتوى يستخدم `fieldsDisplay`.
* صفحة المعاينة الكاملة تعرض `label` أو `labelEn` حسب اتجاه اللغة.

### API

استجابة API ترجع القيم بطريقتين:

* `fields`: القيم الخام المخزنة، مثل رقم المستخدم أو رقم التصنيف أو رقم المحتوى.
* `fieldsDisplay`: كائنات عرض تحتوي على:
  * `value`
  * `label`
  * `labelEn`

مثال:

```json
{
  "fields": {
    "relatedContent": "12"
  },
  "fieldsDisplay": {
    "relatedContent": {
      "value": "12",
      "label": "عنوان عربي",
      "labelEn": "English Title"
    }
  }
}
```

في حالة `MultiSelect` تكون `fieldsDisplay` مصفوفة من نفس الشكل.

## 7. التنسيق والأداء (CSS & Optimization)

* الالتزام التام بمعايير التطوير النظيفة: تم التخلص من كافة التنسيقات المضمنة (`style="..."`) وأكواد الـ `<style>` في واجهة `ContentTypes.cshtml` ونقلها إلى صفوف استباقية محددة في ملف التنسيق الموحد `custom.css` (مثل `.tp-avatar-lg` و `.tp-max-w-200`).
* المحاذاة التلقائية الذكية بناء على اتجاه المستند الحالي دون تدخل من الجافا سكريبت، مما يضمن خفة حركة الواجهة وسلاستها عند التنقل بين العربية والإنجليزية.
